# Спецификация требований к программному обеспечению «SalesTracker»

---

## 1. Введение

### 1.1 Цель

Целью данной спецификации является определение требований к программному обеспечению **SalesTracker** — инструменту автоматизированного сбора и агрегации данных о заказах с маркетплейсов Wildberries и Ozon. Документ предназначен для разработчика продукта и служит основанием для проектирования, реализации и тестирования.

### 1.2 Область применения

Программное обеспечение разрабатывается для использования селлерами (продавцами) маркетплейсов Wildberries и Ozon, которые ведут учёт проданных товаров по артикулам. Продукт автоматически выгружает заказы через официальные API маркетплейсов, подсчитывает количество проданных единиц каждого артикула за выбранный период и отображает результат в виде наглядного отчёта. Это избавляет селлера от ручного учёта продаж в таблицах и блокнотах.

### 1.3 Определения, акронимы и сокращения

- **SRS** (Software Requirements Specification) — спецификация требований к программному обеспечению.
- **API** (Application Programming Interface) — программный интерфейс взаимодействия с внешней системой.
- **Маркетплейс** — торговая онлайн-площадка (в контексте продукта — Wildberries и Ozon).
- **Артикул** — уникальный идентификатор товарной позиции у продавца.
- **Селлер** — продавец, владелец магазина на маркетплейсе.
- **WB** — Wildberries.
- **ФР** — функциональное требование.
- **НФР** — нефункциональное требование.
- **СУБД** — система управления базами данных.
- **MoSCoW** — метод приоритизации требований (Must / Should / Could / Won't have).
- **FBO / FBS** — модели логистики на маркетплейсах (Fulfillment by Operator / by Seller).
- **JDK** (Java Development Kit) — комплект разработчика на языке Java.
- **GUI** (Graphical User Interface) — графический интерфейс пользователя.
- **Swing** — стандартная библиотека Java для создания оконного графического интерфейса.
- **Maven** — система автоматической сборки Java-проектов и управления зависимостями.

## 2. Общее описание

### 2.1 Продукт

Программное обеспечение представляет собой настольное (десктопное) приложение на языке **Java** с простым графическим интерфейсом на библиотеке **Swing**. Приложение подключается к API маркетплейсов по сохранённым ключам, автоматически (по таймеру) и по запросу пользователя выгружает данные о заказах, сохраняет их во встроенной локальной базе данных **H2** и предоставляет пользователю отчёты по продажам с группировкой по дате и артикулу. В продукте отсутствуют учётные записи, логины и пароли — приложение работает локально на устройстве пользователя.

### 2.2 Функции

Программное обеспечение должно иметь следующие функциональные возможности:

- привязка магазинов Wildberries и Ozon через ввод API-ключей;
- автоматическая (по расписанию/таймеру) и ручная синхронизация заказов с маркетплейсов;
- формирование отчёта за выбранный день с разбивкой «артикул → количество»;
- формирование сводного отчёта за произвольный период (до 90 дней);
- фильтрация отчётов по магазину;
- сравнительный анализ продаж за два периода;
- выявление топ-N артикулов и артикулов с отрицательной динамикой;
- экспорт отчётов в форматы XLSX и CSV;
- отображение сообщений об ошибках при недоступности API.

### 2.3 Целевая аудитория

Программное обеспечение предназначено для использования тремя категориями пользователей (выделены методом опорных точек зрения VORD):

- **Селлер** — владелец магазина, выполняет полный цикл работы: привязку магазинов, просмотр и экспорт отчётов;
- **Менеджер** — помощник селлера, формирует ежедневные и периодные отчёты;
- **Аналитик** — анализирует данные «вдолгую»: сравнивает периоды, выявляет тренды.

Косвенно заинтересованными лицами являются бухгалтер селлера, поставщик товара и конечный покупатель.

## 3. Требования

### 3.1 Функциональные требования

Приоритеты указаны по методу MoSCoW: **M** — Must have, **S** — Should have, **C** — Could have.

1. *ФР-01* (M): Пользователь может ввести API-ключ магазина Wildberries через форму настроек.
2. *ФР-02* (M): Система должна сохранить введённый API-ключ Wildberries в локальной БД и отобразить подтверждение.
3. *ФР-03* (M): Пользователь может ввести API-ключ магазина Ozon через форму настроек.
4. *ФР-04* (M): Система должна сохранить введённый API-ключ Ozon в локальной БД и отобразить подтверждение.
5. *ФР-05* (S): Пользователь может удалить привязку магазина; после удаления данные ключа удаляются из локальной БД.
6. *ФР-06* (M): Пользователь может нажать кнопку «Синхронизировать» для принудительного запуска загрузки заказов.
7. *ФР-06а* (M): Система должна автоматически запускать синхронизацию заказов по таймеру через заданный интервал (по умолчанию — 1 раз в час), пока приложение запущено, без участия пользователя.
8. *ФР-07* (M): Система должна отправить запрос к API маркетплейса и сохранить полученные заказы в локальной БД; время выполнения — не более 60 секунд при объёме до 5 000 заказов.
9. *ФР-08* (M): Пользователь может просмотреть отчёт за выбранный день в виде таблицы «артикул — название — маркетплейс — количество заказов»; отчёт строится не дольше 3 секунд при объёме до 10 000 записей в БД.
10. *ФР-09* (M): Пользователь может выбрать в фильтре один магазин, несколько магазинов или «все магазины» перед построением любого отчёта.
11. *ФР-10* (M): Пользователь может сформировать сводный отчёт за период длительностью от 1 до 90 дней с разбивкой по дням и артикулам.
12. *ФР-11* (S): Пользователь может задать два периода равной длительности и получить таблицу сравнения по артикулам с колонками «Период 1», «Период 2», «Δ количество», «Δ %».
13. *ФР-12* (S): Пользователь может получить список топ-N артикулов по количеству продаж за выбранный период, где N задаётся в диапазоне от 5 до 50.
14. *ФР-13* (C): Пользователь может получить список артикулов с отрицательной динамикой продаж между двумя периодами, отсортированный по величине падения в процентах.
15. *ФР-14* (M): Пользователь может выгрузить текущий просматриваемый отчёт в файл формата XLSX.
16. *ФР-15* (S): Пользователь может выгрузить текущий просматриваемый отчёт в файл формата CSV.
17. *ФР-16* (M): Система должна отобразить сообщение с текстом ошибки, если API маркетплейса вернул ошибку или недоступен в течение 15 секунд.

### 3.2 Нефункциональные требования

1. *НФР-01* (Производительность): Построение любого отчёта на объёме до 10 000 записей должно занимать не более 3 секунд.
2. *НФР-02* (Производительность): Синхронизация заказов с одного маркетплейса при объёме до 5 000 заказов должна занимать не более 60 секунд.
3. *НФР-03* (Безопасность): API-ключи маркетплейсов должны храниться в локальной БД в закодированном виде (например, Base64 либо простое симметричное преобразование), а не открытым текстом; файл БД хранится только на устройстве пользователя.
4. *НФР-04* (Надёжность): При сбое внешнего API система должна сохранить ранее загруженные данные и не повредить локальную БД.
5. *НФР-05* (Удобство использования): Формирование дневного отчёта должно выполняться не более чем в 3 клика от главного экрана.
6. *НФР-06* (Переносимость): Приложение должно запускаться на ОС Windows 10/11 и Linux без перекомпиляции — за счёт исполнения в виртуальной машине Java (JVM); требуется только установленная JDK/JRE версии 17 или выше.
7. *НФР-07* (Сопровождаемость): Архитектура должна позволять добавить новый маркетплейс без изменения существующего кода работы с отчётами — через отдельный класс-адаптер, реализующий общий Java-интерфейс (например, `MarketplaceClient`).
8. *НФР-08* (Хранение данных): Локальная БД должна сохранять историю заказов не менее чем за 12 месяцев.

## 4. Интерфейсы

### 4.1 Интерфейс пользователя

Интерфейс пользователя должен быть интуитивно понятным и простым в использовании. Он реализуется средствами библиотеки **Swing** в виде одного главного окна. Основные экраны (вкладки или переключаемые панели): «Настройки» (привязка магазинов), «Отчёт за день», «Отчёт за период», «Аналитика». Навигация между экранами осуществляется через боковое меню или вкладки. Отчёты отображаются в виде таблиц (компонент `JTable`) с возможностью сортировки по столбцам и кнопкой экспорта.

### 4.2 Интерфейс аппаратуры

Программное обеспечение должно корректно взаимодействовать с устройством пользователя (ПК или ноутбук) и не требовать специализированного оборудования. Для работы необходимо подключение к сети Интернет для обращения к API маркетплейсов.

### 4.3 Программные интерфейсы

Программное обеспечение взаимодействует со следующими внешними системами:

- **Wildberries Statistics API** — получение списка заказов (метод выгрузки заказов по дате);
- **Ozon Seller API** — получение отправлений (методы `/v3/posting/fbs/list` и аналоги для FBO);
- **Локальная СУБД H2** — встраиваемая база данных на Java (файловый режим); хранение заказов и агрегатов. Доступ осуществляется через стандартный интерфейс JDBC.

## 5. Требования к производительности

### 5.1 Временные требования

Программное обеспечение должно обеспечивать отклик на запросы пользователя в течение 3 секунд при построении отчётов (на объёме до 10 000 записей) и в течение 60 секунд при синхронизации заказов с маркетплейса (до 5 000 заказов). Сообщение об ошибке недоступности API должно появляться не позднее чем через 15 секунд после запроса.

### 5.2 Нагрузочные требования

Программное обеспечение должно удерживать пиковую нагрузку при работе с несколькими привязанными магазинами одновременно (до 5 магазинов) и хранить историю заказов за период не менее 12 месяцев без деградации скорости построения отчётов.

## 6. Требования к поддержке

### 6.1 Управление изменениями

В случае необходимости внести изменения, процесс изменения программного обеспечения должен быть зафиксирован в системе контроля версий (Git). Каждое изменение требований оформляется в журнале изменений спецификации с указанием версии и даты. Изменение API маркетплейса обрабатывается через обновление соответствующего модуля-адаптера без затрагивания остального кода.

### 6.2 Требования к тестированию

Программное обеспечение должно проходить все необходимые тесты перед выпуском. Каждое функциональное требование должно быть покрыто тест-кейсом. Модули интеграции с API тестируются на корректных и ошибочных ответах (включая таймауты и невалидные ключи).

## 7. Аналоги и зависимости

Аналоги продукта: MPSTATS, Маяк, HunterSales — коммерческие сервисы аналитики маркетплейсов. SalesTracker отличается простотой и бесплатностью, покрывая только базовую задачу учёта продаж по артикулам.

Программные зависимости (подключаются через Maven):

- среда исполнения — **JDK 17+** (Java);
- система сборки и управления зависимостями — **Apache Maven**;
- графический интерфейс — **Swing** (входит в состав JDK, отдельная зависимость не требуется);
- встроенная СУБД — **H2 Database** (артефакт `com.h2database:h2`);
- HTTP-клиент для запросов к API — стандартный `java.net.http.HttpClient` (входит в JDK, без сторонних библиотек);
- работа с JSON-ответами API — библиотека **Jackson** или **Gson** (одна на выбор);
- экспорт в XLSX — библиотека **Apache POI** (артефакт `org.apache.poi:poi-ooxml`);
- экспорт в CSV выполняется штатными средствами Java (запись текста), отдельная библиотека не нужна.

Стек намеренно выбран простым: без веб-фреймворков, без Spring, без ORM. Доступ к БД — через JDBC и обычные SQL-запросы, что подходит для учебного проекта.

## 8. Ограничения

### 8.1 Лицензирование

Программное обеспечение распространяется как свободное ПО и может использоваться другими селлерами. Использование API маркетплейсов осуществляется в соответствии с условиями использования Wildberries и Ozon; пользователь применяет собственные API-ключи.

### 8.2 Технические ограничения

- Продукт зависит от доступности и стабильности API маркетплейсов;
- ограничение по частоте запросов (rate-limit) API Wildberries — необходимо учитывать при синхронизации;
- продукт не хранит и не обрабатывает персональные данные покупателей сверх того, что возвращает API.

---

## 9. Отказы

### 9.1 Устойчивость к отказам

Программное обеспечение должно быть устойчиво к временной недоступности API маркетплейсов: при сбое запроса система сохраняет ранее загруженные данные и информирует пользователя об ошибке (ФР-16, НФР-04), не прерывая работу с уже накопленными отчётами.

### 9.2 Восстановление после отказов

В случае отказа программного обеспечения предусмотрено восстановление работы за счёт сохранения всех данных в локальной БД. После перезапуска приложение продолжает работу с последнего сохранённого состояния. Повторная синхронизация не дублирует уже загруженные заказы (проверка по уникальному идентификатору заказа).
