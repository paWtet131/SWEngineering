# Описание архитектуры SalesTracker

Документ для разработчика: структура проекта, назначение модулей и классов,
ключевые решения. Соответствует разделу 10 спецификации.

---

## 1. Общая структура

Приложение — однооконное настольное приложение на Java + Swing.
Код разделён на слои:

```
интерфейс (ui)  →  сервисы (service)  →  доступ к данным (db) / API (marketplace)
                                   ↘  модели (model), утилиты (util)
```

- **ui** — экраны и компоненты Swing;
- **service** — бизнес-логика (синхронизация, построение отчётов, экспорт);
- **db** — доступ к базе H2 через JDBC (классы-DAO);
- **marketplace** — адаптеры внешних API маркетплейсов;
- **model** — простые классы данных;
- **util** — вспомогательные функции.

Слой интерфейса обращается к сервисам, сервисы — к DAO и адаптерам.
Обратных зависимостей нет: например, `db` ничего не знает про `ui`.


## 3. Модель данных (БД H2)

База создаётся автоматически в файле `data/salestracker.mv.db`.

**Таблица `shops`** — привязанные магазины:

| Поле | Тип | Описание |
| --- | --- | --- |
| id | INT, PK, auto | идентификатор |
| marketplace | VARCHAR | WB или OZON |
| name | VARCHAR | название магазина |
| api_key | VARCHAR | API-ключ (закодирован Base64) |
| client_id | VARCHAR | Client-Id (только для Ozon) |

**Таблица `orders`** — заказы:

| Поле | Тип | Описание |
| --- | --- | --- |
| order_id | VARCHAR | уникальный id заказа на маркетплейсе |
| shop_id | INT | ссылка на магазин |
| marketplace | VARCHAR | WB или OZON |
| article | VARCHAR | артикул продавца |
| name | VARCHAR | название товара |
| order_date | DATE | дата заказа |
| quantity | INT | количество единиц |

Первичный ключ `(order_id, shop_id)`. Сохранение выполняется через `MERGE`,
поэтому повторная синхронизация не создаёт дубликатов.

## 4. Ключевые архитектурные решения

### Добавление нового маркетплейса (НФР-07)
Все адаптеры реализуют интерфейс `MarketplaceClient` с единственным методом
`fetchOrders(shop, from, to)`. Чтобы подключить новый маркетплейс, нужно:
1. добавить значение в `enum Marketplace`;
2. создать новый класс, реализующий `MarketplaceClient`;
3. добавить его в `switch` метода `SyncService.clientFor()`.

Код отчётов, хранения и интерфейса при этом не меняется.

### Единый формат отчёта
Все отчёты `ReportService` возвращают объект `ReportTable` (заголовки + строки).
Это позволяет одинаково показывать их в `JTable` и экспортировать
в XLSX/CSV без дублирования кода.

### Работа с потоками
- Ручная синхронизация запускается в `SwingWorker`, чтобы не «замораживать» окно.
- Автосинхронизация работает в отдельном фоновом потоке-демоне (`SyncScheduler`).
- Результаты из фоновых потоков выводятся в интерфейс через
  `SwingUtilities.invokeLater`.

### Безопасность ключей (НФР-03)
API-ключи кодируются классом `KeyCodec` (Base64) перед записью в БД
и раскодируются при чтении. Открытым текстом в файле БД они не хранятся.

### Обработка ошибок API (ФР-16, НФР-04)
Адаптеры бросают `MarketplaceException` при недоступности, таймауте (15 секунд)
или ошибочном ответе. Интерфейс показывает текст ошибки в диалоговом окне,
при этом ранее загруженные данные в БД остаются нетронутыми.

## 5. Сборка и зависимости

Сборка — Maven (`pom.xml`). Используется `maven-shade-plugin` для создания
единого исполняемого jar со всеми зависимостями.

| Зависимость | Назначение |
| --- | --- |
| `com.h2database:h2` | встроенная БД |
| `com.google.code.gson:gson` | разбор JSON-ответов API |
| `org.apache.poi:poi-ooxml` | экспорт в XLSX |

HTTP-запросы выполняются штатным `java.net.http.HttpClient` (входит в JDK),
экспорт в CSV — штатной записью текста. Сторонних веб-фреймворков, Spring
и ORM нет — доступ к БД через чистый JDBC и SQL.
