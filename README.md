## 1. Описание проблемы

**Проблема:** селлеры маркетплейсов вручную ведут ежедневный учёт заказанных товаров по артикулам (в Excel, блокноте или просто в голове), чтобы понимать динамику продаж, планировать закупки и поставки на склад. Личные кабинеты Wildberries и Ozon показывают заказы списком, но не дают удобной агрегированной сводки «сколько штук каждого артикула продано за конкретный день» — особенно если работаешь сразу на двух площадках.

**Воздействует на:**

- селлеров маркетплейсов (ИП и самозанятых), торгующих собственным товаром;
- менеджеров, нанятых селлерами для ведения учёта;
- бухгалтеров и аналитиков, которым нужны сводные данные.

**Результатом чего является:**

- потеря времени на ручное копирование данных из ЛК (1–2 часа в день);
- ошибки в подсчётах из-за человеческого фактора;
- опоздания с поставками на склад из-за того, что неочевидно, какие позиции расходятся быстрее;
- невозможность сравнить продажи одной и той же позиции на WB и Ozon в едином окне;
- упущенная выручка из-за out-of-stock по ходовым артикулам.

**Выигрыш от ПП:**

- автоматический сбор заказов с WB и Ozon через официальные API;
- сводный отчёт за выбранную дату: артикул → количество заказов;
- единая статистика по обоим маркетплейсам в одном приложении;
- экономия 1–2 часов рабочего времени селлера ежедневно;
- точность данных (исключение ручного ввода);
- история продаж за любой период для аналитики и планирования.

## 2. Назначение программного средства

**Какие системы смогут применять ПС для решения своих задач:**
настольное (десктопное) приложение на Java с простым графическим интерфейсом (Swing), запускаемое локально на компьютере пользователя. Селлер открывает приложение SalesTracker, привязывает API-ключи своих магазинов на Wildberries и Ozon и получает сводные отчёты по продажам. Учётных записей, логинов и паролей нет — приложение работает локально.

**От каких систем/пользователей поступают входные данные ПС:**

- API Wildberries (Statistics API — метод получения заказов);
- API Ozon Seller (методы `/v3/posting/fbs/list` и аналоги для FBO);
- пользователь (вводит API-ключи, выбирает даты для отчёта).

**Кто/что использует результаты:**

- селлер — просматривает отчёты в интерфейсе приложения;
- менеджер селлера — использует то же приложение для формирования отчётов;
- внешние системы — экспорт в Excel/CSV для дальнейшей работы в 1С или Google Sheets.

**Какие сторонние системы/сервисы участвуют в процессе обработки данных:**

- API Wildberries;
- API Ozon.

### Роли пользователей и их потребности

| Роль | Потребности |
| --- | --- |
| **Селлер** | Видеть, сколько и каких артикулов продано за конкретный день по всем своим магазинам; экспортировать отчёт; управлять API-ключами магазинов |
| **Менеджер** | Формировать ежедневные и периодные отчёты по магазинам селлера; делать выгрузки для бухгалтера и поставщика |
| **Аналитик** | Смотреть данные «вдолгую»: сравнивать продажи за разные периоды, выявлять топ-артикулы и проседающие позиции, искать тренды для планирования закупок |

> **Примечание.** Роли разделяются не по правам доступа (учётных записей в приложении нет), а по характеру решаемых задач: одни и те же люди могут запускать одно и то же приложение для разных целей.

**Пояснение к секторам диаграммы:**

- **Сверху — Основные системы и бизнес-процессы (Used by).** Бизнес-процессы селлера, ради которых существует наша система: ежедневный учёт продаж, планирование закупок и поставок, формирование отчётности для бухгалтера и владельца бизнеса.
- **Слева — Заинтересованные лица (Uses).** Три роли пользователей: Селлер, Менеджер, Аналитик. Связь двунаправленная: пользователь даёт ввод (API-ключи, выбор даты, фильтры), система возвращает результат (отчёт, экспорт).
- **Справа — Сторонние системы и сервисы (Uses).** Внешние API маркетплейсов: Wildberries Statistics API и Ozon Seller API. Они поставляют данные о заказах.
- **Снизу — Подчинённые системы (Depends on).** Технические подсистемы, на которых работает SalesTracker: встроенная СУБД H2 для хранения заказов и агрегатов и внутренний планировщик задач (на основе `java.util.Timer` / `ScheduledExecutorService`) для автоматического опроса API маркетплейсов по расписанию.

## 4. Анализ конкурентов

В качестве конкурентов рассмотрены три сервиса аналитики маркетплейсов, ориентированные на тот же сегмент рынка — обычных пользователей (`U`) и микрокомпании до 10 сотрудников (`XS`):

- **MPSTATS** — крупный аналитический сервис для WB и Ozon;
- **Маяк** — расширение для браузера и сервис аналитики Wildberries;
- **HunterSales** — сервис аналитики продаж для селлеров WB и Ozon.

### Сравнительная таблица

| Проблема | Сегмент рынка | MPSTATS | Маяк | HunterSales | SalesTracker | Приоритет |
| --- | --- | :---: | :---: | :---: | :---: | :---: |
| Автоматический сбор заказов с WB через API | U, XS | + | + | + | + | e |
| Автоматический сбор заказов с Ozon через API | U, XS | + | − | + | + | e |
| Ежедневный отчёт «артикул → количество за дату» | U, XS | + | + | + | + | e |
| Единое приложение для WB и Ozon | U, XS | + | − | + | + | e |
| Простой интерфейс без перегруза функциями | U | − | − | − | + | e |
| Бесплатное использование для одного селлера | U, XS | − | − | − | + | u |
| Глубокая аналитика конкурентов и ниш | XS | + | + | + | − | d |
| Расчёт юнит-экономики и прибыли | XS | + | − | + | − | d |
| Экспорт отчётов в Excel/CSV | U, XS | + | + | + | + | u |
| Мобильное приложение | U | + | + | − | − | d |

**Условные обозначения:**

- Сегмент рынка: `U` — обычные домашние пользователи, `XS` — микрокомпании до 10 сотрудников.
- Наличие функции: `+` — есть, `−` — нет.
- Приоритет: `e` (essential) — обязательно, `u` (useful) — полезно, `d` (desirable) — желательно.

### Вывод по анализу конкурентов

Существующие решения — это мощные коммерческие платформы (от 5 000 до 20 000 ₽/мес), ориентированные на аналитику ниш, конкурентов и юнит-экономику. Они избыточны для рядового селлера, которому нужна базовая задача: видеть, сколько штук каждого артикула продано за день.

**Полезные идеи, которые стоит взять из конкурентов:**

- объединение данных нескольких маркетплейсов в одном интерфейсе (есть у MPSTATS и HunterSales);
- экспорт отчётов в Excel (есть у всех троих) — пригодится для отправки бухгалтеру;
- группировка по дате с возможностью выбрать диапазон.

**Ниша SalesTracker:** простой бесплатный инструмент для базового учёта, без сложной аналитики и низким порогом входа. Для селлера, который ведёт всё в блокноте, наш продукт станет первым шагом к автоматизации без необходимости разбираться в «профессиональных» платформах.

## 5. Описание продукта по шаблону Moore

> *Шаблон: Moore, Geoffrey A. (1991). Crossing the Chasm: Marketing and Selling High-Tech Products to Mainstream Customers. New York: HarperBusiness.*

**Для** селлеров маркетплейсов Wildberries и Ozon (ИП, самозанятых и микрокоманд),

**которые** ежедневно тратят 1–2 часа на ручной подсчёт проданных артикулов и нуждаются в простой автоматизации учёта без сложной аналитики,

**этот** SalesTracker

**является** настольным приложением для автоматизированного сбора и агрегации заказов с маркетплейсов,

**который** автоматически выгружает заказы через официальные API, считает количество проданных штук по каждому артикулу за выбранный день и показывает результат в виде наглядного отчёта,

**в отличие от** MPSTATS, Маяк и HunterSales — дорогих профессиональных платформ со сложным интерфейсом, ориентированных на глубокую аналитику конкурентов,

**наш продукт** делает только одно, но делает это просто и бесплатно — даёт селлеру за пару кликов готовую сводку «сколько и каких артикулов продано сегодня» по всем его магазинам сразу.

## 1. Иерархия заинтересованных лиц (метод опорных точек зрения)

Для выделения категорий пользователей использован метод **VORD** (Viewpoint-Oriented Requirements Definition). Согласно методу, все заинтересованные лица делятся на две большие группы:

- **Прямые** опорные точки зрения — пользователи и системы, которые непосредственно взаимодействуют с продуктом, получают от него сервисы, посылают ему данные и управляющие сигналы;
- **Косвенные** опорные точки зрения — лица и организации, которые сами с системой не работают, но заинтересованы в результатах её функционирования.

Дополнительно выделены **системные** и **организационные** точки зрения, влияющие на требования к продукту.

> **Особенность продукта.** SalesTracker — простое локальное приложение, в котором отсутствуют учётные записи, логины и пароли. Поэтому роли разделяются не по правам доступа, а по характеру решаемых задач: одни и те же люди могут работать с одним и тем же приложением, но использовать его для разных целей.

## 2. Диаграмма вариантов использования

На диаграмме показаны три прямых актора, граница системы и доступные каждому из них прецеденты (use case). Прецедент **«Загрузить заказы»** связан отношением `«include»` со всеми прецедентами формирования отчётов: каждый раз при открытии отчёта система при необходимости обращается за свежими данными к внешним API. Прецедент **«Запустить синхронизацию»** связан с «Загрузить заказы» отношением `«extend»` — это опциональное ручное расширение, доступное селлеру. Кроме того, встроенный **Планировщик** (системный актор) периодически инициирует прецедент «Загрузить заказы» автоматически по расписанию, без участия пользователя (автоматическая синхронизация).

<img width="800" height="696" alt="usecase" src="https://github.com/user-attachments/assets/2b7ae719-a9aa-46c1-9476-65489e97b6f0" />


**Пояснение к связям:**

- **«include»** — прецедент «Загрузить заказы» обязательно вызывается из прецедентов просмотра отчётов, если данные в локальной БД устарели.
- **«extend»** — прецедент «Запустить синхронизацию» расширяет «Загрузить заказы»: селлер может принудительно вызвать выгрузку, не дожидаясь автоматической.
- Внешние системы **API Wildberries** и **API Ozon** обозначены как `«API»`-актор — он обеспечивает входные данные для прецедента «Загрузить заказы».
- Внутренний **Планировщик** также выступает `«API»`-актором: по таймеру он запускает прецедент «Загрузить заказы», обеспечивая автоматическую синхронизацию даже без действий пользователя.

## 3. Сценарии использования (use case)

### 3.1. Селлер — «Просмотреть отчёт за день»

**Название:** Просмотреть отчёт за день
**Актор:** Селлер
**Предусловие:**

- Приложение SalesTracker запущено.
- К приложению привязан хотя бы один магазин (введены валидные API-ключи Wildberries и/или Ozon).
- Локальная БД содержит данные о заказах (планировщик хотя бы один раз успешно выгрузил заказы, либо синхронизация была запущена вручную).

**Основной поток:**

1. Селлер открывает раздел «Отчёт за день» в интерфейсе SalesTracker.
2. Система отображает форму выбора параметров отчёта (дата, магазин).
3. Селлер выбирает интересующую дату (по умолчанию — текущая) и магазин (по умолчанию — все).
4. Селлер нажимает кнопку «Сформировать».
5. Система запрашивает агрегированные данные из локальной БД за указанную дату.
6. Система отображает таблицу: артикул, название товара, маркетплейс, количество заказов.
7. В нижней строке таблицы система выводит итоговое количество проданных штук за день.

**Альтернативные потоки:**

- **3а.** За выбранную дату нет данных:
  - 3а.1. Система отображает сообщение «За выбранную дату продаж не зафиксировано».
  - 3а.2. Сценарий завершается.
- **5а.** Данные в БД устарели (последняя выгрузка более 24 часов назад):
  - 5а.1. Система отображает предупреждение и предлагает запустить ручную синхронизацию.
  - 5а.2. При согласии селлера выполняется прецедент «Загрузить заказы».
  - 5а.3. После завершения выгрузки возврат к шагу 5 основного потока.

**Постусловие:**

- Селлер увидел корректный отчёт за указанную дату.
- При необходимости локальная БД дополнилась свежими заказами с маркетплейсов.

### 3.2. Менеджер — «Сформировать отчёт за период»

**Название:** Сформировать отчёт за период
**Актор:** Менеджер
**Предусловие:**

- Приложение SalesTracker запущено и настроено селлером (магазины привязаны).
- Локальная БД содержит данные о заказах за интересующий период.

**Основной поток:**

1. Менеджер открывает раздел «Отчёты» в интерфейсе SalesTracker.
2. Менеджер выбирает диапазон дат (например, 01.05.2026 — 27.05.2026) и один или несколько магазинов.
3. Менеджер нажимает кнопку «Сформировать».
4. Система агрегирует данные за указанный период из локальной БД.
5. Система отображает сводную таблицу с разбивкой по дням и артикулам.
6. Менеджер нажимает «Экспорт в Excel».
7. Система формирует файл XLSX и предлагает его сохранить.

**Альтернативные потоки:**

- **2а.** Указан период длиннее 90 дней:
  - 2а.1. Система отображает предупреждение «Максимальный период — 90 дней».
  - 2а.2. Возврат к шагу 2.
- **6а.** Менеджер не нажимает «Экспорт»:
  - 6а.1. Сценарий завершается после шага 5 (отчёт просмотрен только на экране).

**Постусловие:**

- Менеджер получил сводный отчёт за период.
- При экспорте файл XLSX сохранён на устройство.

### 3.3. Аналитик — «Выявить топ-артикулы и сравнить периоды»

**Название:** Выявить топ-артикулы и сравнить периоды
**Актор:** Аналитик
**Предусловие:**

- Приложение SalesTracker запущено.
- В локальной БД накоплены данные о заказах минимум за два сопоставимых периода (например, два месяца подряд).

**Основной поток:**

1. Аналитик открывает раздел «Аналитика» в интерфейсе SalesTracker.
2. Аналитик задаёт два сравниваемых периода (например, «текущий месяц» и «предыдущий месяц»).
3. Аналитик выбирает магазины, по которым проводится анализ.
4. Аналитик нажимает кнопку «Сравнить».
5. Система запрашивает агрегированные данные из локальной БД за оба периода.
6. Система отображает таблицу с топ-N артикулами и колонками: «Артикул», «Период 1», «Период 2», «Δ количество», «Δ %».
7. Аналитик сортирует таблицу по столбцу «Δ %» по убыванию, чтобы увидеть растущие позиции.
8. Аналитик сортирует таблицу по столбцу «Δ %» по возрастанию, чтобы увидеть проседающие позиции.
9. Аналитик делает выводы о трендах и при необходимости экспортирует результат в Excel.

**Альтернативные потоки:**

- **5а.** Во втором периоде нет данных:
  - 5а.1. Система отображает таблицу только по первому периоду с пометкой «Сравнение невозможно: во втором периоде нет данных».
  - 5а.2. Сценарий завершается.
- **6а.** Аналитик ограничивает выборку конкретным маркетплейсом:
  - 6а.1. Возврат к шагу 3 с уточнёнными параметрами.

**Постусловие:**

- Аналитик получил сравнительную картину продаж и список топ- и анти-топ артикулов.
- При экспорте файл XLSX сохранён на устройство.

## 4. Проверка качества требований

Каждое требование проверено по четырём правилам формулировки:

| Правило | Как соблюдается в перечне |
| --- | --- |
| **Один глагол** | Каждое требование описывает одно действие. Бывшее FR-01 v1.0 («ввести и сохранить») разделено на FR-01 и FR-02 в v2.0. |

| **Атомарность** | Ни одно требование не содержит «и»/«или» в значении множественного действия. Экспорт разделён на FR-14 (XLSX) и FR-15 (CSV). |

| Конфликтующие требования | Причина конфликта | Возможное разрешение |
| --- | --- | --- |
| FR-07: «не более 60 секунд при объёме до 5 000 заказов» и FR-08: «не дольше 3 секунд при объёме до 10 000 записей» | Нет конфликта — разные операции: FR-07 — сетевой запрос к внешнему API (ограничен скоростью интернета и rate-limit маркетплейса), FR-08 — локальный запрос к встроенной СУБД H2 (ограничен только диском) | — |
| FR-14 (XLSX) и FR-15 (CSV) | Потенциальный конфликт по сложности реализации: XLSX требует внешнюю библиотеку (Apache POI), CSV пишется штатными средствами Java | Конфликт минимальный. XLSX — Must have (основной формат для бухгалтера), CSV — Should have (запасной вариант). При нехватке времени CSV реализуется первым (проще), XLSX — вторым |
| FR-10 (период до 90 дней) и FR-11 (сравнение двух периодов) | При сравнении двух периодов по 90 дней каждый объём данных удваивается. На больших объёмах время построения отчёта может превысить 3 секунды | FR-11 имеет приоритет Should, а не Must. При больших объёмах допустимо увеличить время до 5 секунд — это нефункциональное уточнение |

**Итого:** серьёзных конфликтов реализуемости не выявлено. Все требования реализуемы стандартным стеком (Java + Maven + H2 + `HttpClient` + Apache POI).

### А6. SWOT-анализ спецификации

| Квадрант | Ответ применительно к спецификации SalesTracker |
| --- | --- |
| **Strengths (Сильные стороны)** | Требования атомарны, каждое привязано к конкретной роли и use case. Есть измеримые критерии производительности (секунды, объёмы). Приоритеты расставлены по MoSCoW — разработчик знает, с чего начинать. Нет стоп-слов и субъективных оценок. |
| **Weaknesses (Слабые стороны)** | На момент анализа нефункциональные требования не были описаны отдельным блоком (безопасность хранения API-ключей, кроссплатформенность, объём БД) — впоследствии они вынесены в отдельный раздел итоговой спецификации (НФР-01…НФР-08). Нет детальных макетов интерфейса (описана лишь структура экранов Swing). Обработка ошибок описана одним FR-16, в реальности нужна более детальная проработка (retry-логика, учёт rate-limit API). |
| **Opportunities (Возможности)** | Спецификация легко расширяется: добавление третьего маркетплейса (Яндекс Маркет) потребует только нового FR на ввод/сохранение ключа и расширение FR-07. Аналитический блок (FR-11–FR-13) при развитии может стать отдельным модулем с визуализацией (графики). |
| **Threats (Угрозы)** | API маркетплейсов могут измениться без предупреждения — это внешний риск, не покрытый требованиями. Rate-limit API Wildberries (100 запросов в минуту) может быть превышен при большом количестве магазинов — потенциальный сбой FR-07. Хранение API-ключей в открытом виде в файле БД H2 — угроза безопасности; в спецификации она частично закрыта требованием хранить ключи в закодированном виде (Base64/простое преобразование). |
